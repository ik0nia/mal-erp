State Machines — Modulul Achiziții

Documentul descrie complet automatele de stări (state machines) pentru cele patru entități principale ale modulului de achiziții: PurchaseRequest, PurchaseOrder, PurchaseRequestItem și PurchaseOrderItem. Sunt documentate și algoritmul FIFO de alocare cantități, logica needsApproval(), algoritmul recalculateStatus() și toate notificările emise.

4. Algoritmul FIFO de alocare cantități (afterCreate)

Când un PurchaseOrder este creat, metoda afterCreate() din CreatePurchaseOrder leagă fiecare PurchaseOrderItem de sursele sale (PurchaseRequestItem-uri) din sources_json. Algoritmul alocă cantitățile parțial sau complet, respectând ordinea FIFO (urgente întâi, apoi data limită crescătoare).

4.1 Preprocesare — buildItemsForSupplier()

Înainte de creare, dacă PO-ul este inițiat din BuyerDashboard cu parametrul ?supplier_id=X, formularul este pre-populat de buildItemsForSupplier():

  1. Selectează toate PurchaseRequestItem-urile cu:
    • supplier_id = $supplierId
    • status = pending
    • PNR-ul părinte în status submitted sau partially_ordered
    • quantity > COALESCE(ordered_quantity, 0) (exclude complet comandate)
  2. Sortează: is_urgent DESC, needed_by ASC (FIFO).
  3. Grupează după woo_product_id (sau product_name pentru produse fără WooID).
  4. Pentru fiecare grup: sumează cantitățile rămase (quantity - ordered_quantity), construiește un array sources cu câmpurile: request_item_id, request_number, consultant, location, quantity (cantitate rămasă), is_urgent, needed_by, client_reference, allocated_qty=0.0.
  5. Serialzează sources ca JSON în câmpul sources_json al formularului.

4.2 Execuția afterCreate() — pas cu pas

Întreaga logică este învelită în DB::transaction() cu lockForUpdate() pe fiecare PurchaseRequestItem (previne race condition la creare simultană de PO-uri). 

// Pseudocod simplificat al afterCreate()
foreach ($this->record->items as $orderItem) {
    $sources = json_decode($orderItem->sources_json);  // array de surse

    // 1. FIFO sort: urgent first, then earliest needed_by
    usort($sources, fn($a, $b) =>
        ($b['is_urgent'] - $a['is_urgent'])
        ?: strcmp($a['needed_by'], $b['needed_by'])
    );

    $remaining = (float) $orderItem->quantity;  // cantitatea totală a liniei PO

    foreach ($sources as $source) {
        $requestItem = PurchaseRequestItem::lockForUpdate()->find($source['request_item_id']);

        // Calculăm cât putem aloca din această sursă
        $alreadyOrdered  = $requestItem->ordered_quantity;
        $availableInItem = $requestItem->quantity - $alreadyOrdered;  // max disponibil în sursă
        $canAllocate     = min($remaining, $source['quantity'], $availableInItem);

        if ($canAllocate <= 0) { ... skip ... }

        $remaining -= $canAllocate;

        // Incrementare atomică ordered_quantity
        PurchaseRequestItem::where('id', $requestItem->id)
            ->increment('ordered_quantity', $canAllocate);

        $requestItem->refresh();  // actualizăm valorile post-increment

        if ($requestItem->isFullyOrdered()) {
            // Item complet acoperit → STATUS_ORDERED
            $requestItem->update([
                'status'                 => PurchaseRequestItem::STATUS_ORDERED,
                'purchase_order_item_id' => $orderItem->id,
            ]);
        } else {
            // Item parțial comandat → rămâne PENDING, dar cu referința PO
            $requestItem->update([
                'purchase_order_item_id' => $orderItem->id,
            ]);
        }

        $source['allocated_qty'] = $canAllocate;  // înregistrăm cât s-a alocat
    }

    // Salvăm sources_json cu allocated_qty completat
    $orderItem->updateQuietly([
        'sources_json'             => json_encode($updatedSources),
        'purchase_order_item_id'   => $primaryRequestItem,
    ]);
}

// Recalculăm statusul fiecărui PurchaseRequest afectat (în afara tranzacției)
foreach (array_unique($affectedRequestIds) as $requestId) {
    PurchaseRequest::find($requestId)->recalculateStatus();
}

4.5 Protecție race condition

Crearea simultană a două PO-uri pentru același furnizor ar putea supra-aloca cantitățile. Protecția se face pe trei niveluri:

  1. DB::transaction() — toate incrementările și actualizările sunt atomice.
  2. lockForUpdate() pe PurchaseRequestItem — blochează rândul DB pentru citire și scriere până la finalul tranzacției.
  3. Recalcul post-lock — alreadyOrdered se citește după obținerea lock-ului ($requestItem->ordered_quantity la momentul lock-ului), nu din datele formularului.
  4. firstOrCreate() în getOrCreateDraft() — previne duplicate de drafturi per user.

5. Logica needsApproval()

Metoda PurchaseOrder::needsApproval() este apelată în acțiunea "Plasează comanda" pentru a decide dacă PO-ul se aprobă automat (→ approved) sau necesită aprobare manuală (→ pending_approval). 

public function needsApproval(): bool
{
    $this->loadMissing(['supplier', 'items']);

    // Regula 1: valoare totală PO > plafon furnizor
    if ($this->supplier->po_approval_threshold &&
        (float) $this->total_value > (float) $this->supplier->po_approval_threshold) {
        return true;
    }

    // Regula 2: oricare item depășește cantitatea maximă per produs
    $productIds = $this->items->pluck('woo_product_id')->filter()->unique()->values();

    if ($productIds->isEmpty()) {
        return false;
    }

    // Query unic pentru toate ProductSupplier-urile relevante (evitare N+1)
    $productSupplierMap = ProductSupplier::query()
        ->where('supplier_id', $this->supplier_id)
        ->whereIn('woo_product_id', $productIds)
        ->get(['woo_product_id', 'po_max_qty'])
        ->keyBy('woo_product_id');

    foreach ($this->items as $item) {
        if (! $item->woo_product_id) continue;

        $ps = $productSupplierMap->get($item->woo_product_id);

        if ($ps && $ps->po_max_qty && (float) $item->quantity > (float) $ps->po_max_qty) {
            return true;
        }
    }

    return false;
}

6. Algoritmul recalculateStatus() pe PurchaseRequest

Metoda PurchaseRequest::recalculateStatus() este apelată de fiecare dată când starea unui PurchaseRequestItem sau ordered_quantity se modifică. Folosește saveQuietly() pentru a evita declanșarea evenimentelor Eloquent recurente. 

public function recalculateStatus(): void
{
    $items = $this->items()->get(['status', 'quantity', 'ordered_quantity']);

    // Guard: fără items → nu face nimic
    if ($items->isEmpty()) {
        return;
    }

    // Excludem itemii anulați din calcul
    $active = $items->where('status', '!=', PurchaseRequestItem::STATUS_CANCELLED);

    // Branch 1: toți itemii sunt anulați
    if ($active->isEmpty()) {
        $this->forceFill(['status' => self::STATUS_CANCELLED])->saveQuietly();
        return;
    }

    $total = $active->count();

    // Un item "complet comandat" = ordered_quantity >= quantity
    $fullyOrdered = $active->filter(
        fn ($i) => (float) $i->ordered_quantity >= (float) $i->quantity
    )->count();

    // "Are cel puțin un item parțial comandat" = ordered_quantity > 0
    $hasAnyOrdered = $active->filter(
        fn ($i) => (float) $i->ordered_quantity > 0
    )->count() > 0;

    // Branch 2: toți itemii activi sunt complet acoperiți
    if ($fullyOrdered >= $total) {
        $this->forceFill(['status' => self::STATUS_FULLY_ORDERED])->saveQuietly();

    // Branch 3: cel puțin un item are ordered_quantity > 0 (dar nu toți sunt complet)
    } elseif ($hasAnyOrdered) {
        $this->forceFill(['status' => self::STATUS_PARTIALLY_ORDERED])->saveQuietly();

    // Branch 4: niciun item nu are cantitate comandată → revenire la submitted
    } else {
        $this->forceFill(['status' => self::STATUS_SUBMITTED])->saveQuietly();
    }
}

6.1 Arbore de decizie

Condiție evaluată Status rezultat Cazuri tipice
$items.isEmpty() Fără modificare (return timpuriu) PNR fără niciun item (caz degenerat)
$active.isEmpty() (toți itemii = cancelled) cancelled Consultantul a anulat manual toți itemii
$fullyOrdered >= $total (toți activi: ordered_qty ≥ qty) fully_ordered PO creat și trimis acoperă tot necesarul
$hasAnyOrdered = true (cel puțin un item parțial sau complet acoperit) partially_ordered PO acoperă o parte din itemi/cantități; restul rămâne în așteptare
Niciuna din cele de mai sus (toți activi cu ordered_qty = 0) submitted PO anulat/respins → toate alocările revenite; PNR reintră în coadă

6.2 Contexte de apel

  • afterCreate() din CreatePurchaseOrder — după alocarea FIFO, pentru fiecare PNR afectat
  • markRequestItemsAsOrdered() — la tranzitia PO la sent
  • revertRequestItemsToPending() — la CANCELLED sau REJECTED al PO-ului
  • revertShortfallToRequestItems() — la recepție parțială cu lipsuri
  • cancel_item action din infolist PurchaseRequest — la anularea manuală a unui item

Toate notificările implementează via() = ['database', 'mail'] — sunt trimise simultan în baza de date (vizibile în clopotul Filament) și prin email.

7.1 Detalii suplimentare per notificare

PurchaseRequestSubmittedNotification
  • Logica de selectare destinatari: se iau mai întâi buyers care gestionează explicit furnizorii din necesar (relația User::managedSuppliers). Dacă aceștia lipsesc, fallback la toți userii cu role = ROLE_MANAGER_ACHIZITII.
  • Badge DB: icon heroicon-o-clipboard-document-list, culoare warning dacă există urgente, altfel info.
PurchaseOrderNeedsApprovalNotification
  • Logica de selectare destinatari: query cu whereIn('role', [MANAGER, DIRECTOR_FINANCIAR, DIRECTOR_VANZARI]) orWhere('is_super_admin', true), exclude userul curent.
  • Badge DB: icon heroicon-o-clock, culoare warning.
  • Link acțiune: /app/purchase-orders/{id} — pagina ViewPurchaseOrder unde sunt vizibile butoanele "Aprobă" și "Respinge" (dacă rolul este suficient).
PurchaseOrderRejectedNotification
  • Transmite: $rejectedByName — numele celui care a respins, afișat în email.
  • Efecte anterioare notificării: revertRequestItemsToPending() este apelat înainte de update(['status' => REJECTED]), astfel request items sunt deja revenite la pending la momentul trimiterii notificării.
  • Badge DB: icon heroicon-o-x-circle, culoare danger.
PurchaseOrderReceivedPartialNotification
  • Destinatari: colectați din affectedRequestIds (PNR-urile cu items returnate) → PurchaseRequest.user_id → deduplicare → exclude userul curent.
  • shortfallProducts: array de product_name string-uri; în email se afișează max 5, urmate de "și altele" dacă sunt mai multe.
  • Badge DB: icon heroicon-o-exclamation-triangle, culoare warning.

8. PurchaseOrderItem — Lifecycle și calcule automate

PurchaseOrderItem nu are o state machine proprie, dar conține logică importantă în booted():

Hook Eloquent Acțiune Formulă
saving Recalculare automată line_total line_total = max(0, quantity) × max(0, unit_price), format decimal:4
saved Declanșare PurchaseOrder::recalculateTotals() total_value = SUM(line_total) pe toate itemele PO-ului
deleted Declanșare PurchaseOrder::recalculateTotals() Idem — totalul PO se recalculează și la ștergere item

8.1 Câmpul received_quantity

Câmpul received_quantity (nullable decimal:3) este completat în acțiunea "Recepție marfă". Default în UI: egal cu cantitatea comandată (quantity). Dacă received_quantity < quantity:

  • UI afișează "X / Y ⚠" cu badge warning
  • Shortfall-ul (quantity - received_quantity) este returnat în coada de cumpărare
  • Coloana reception_status din tabel devine "Parțială ⚠"
