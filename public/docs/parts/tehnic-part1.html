Structura aplicației

app/
├── Console/Commands/       # Comenzi Artisan (BI, email, WooCommerce, achiziții, social)
├── Filament/
│   ├── App/                # Panelul principal "App" (utilizatori operaționali)
│   │   ├── Pages/          # Pagini custom (dashboard, rapoarte, chat, social, achiziții)
│   │   ├── Resources/      # CRUD resources Filament
│   │   ├── Search/         # AppGlobalSearchProvider
│   │   └── Widgets/        # Widgeturi dashboard
│   └── Pages/              # Pagini panel Admin (AppSettingsPage, AiUsageLogPage)
├── Http/
│   └── Controllers/        # ChatController, GraphicTemplateController etc.
├── Jobs/                   # Jobs asincrone (email, social, BI, WooCommerce sync)
├── Models/                 # Modele Eloquent (~42 modele)
├── Notifications/          # Notificări Laravel (achiziții)
├── Providers/
│   ├── AppServiceProvider.php
│   └── Filament/AppPanelProvider.php
└── Services/               # Servicii business (AI, Email, SocialMedia, WooCommerce)

database/migrations/        # ~90 fișiere migrații
routes/
├── web.php                 # Rute web principale
├── console.php             # Comenzi schedule
├── chat.php                # Rute chatbot public (fără CSRF/auth)
└── webhooks.php            # Rute webhook (fără CSRF/session/auth)

Gestionare erori

Erorile HTTP 403 și 404 pentru utilizatori autentificați (browsere, non-XHR, non-admin path) sunt redirecționate la pagina filament.app.pages.error-page cu parametrul code.

Algoritmul recalculateStatus()

1. Preia toți itemii (status, quantity, ordered_quantity)
2. active = itemii cu status != cancelled
3. Dacă active.isEmpty() → STATUS_CANCELLED (saveQuietly)
4. fullyOrdered = active where ordered_quantity >= quantity (count)
5. hasAnyOrdered = active where ordered_quantity > 0 (exists)
6. Dacă fullyOrdered >= total → STATUS_FULLY_ORDERED
7. Altfel dacă hasAnyOrdered → STATUS_PARTIALLY_ORDERED
8. Altfel → STATUS_SUBMITTED

Logica needsApproval()

Determină dacă un PO trebuie să treacă prin aprobare înainte de a fi trimis furnizorului.

PurchaseOrder::needsApproval(): bool

1. Regula 1 — Plafon valoric:
   IF supplier.po_approval_threshold IS NOT NULL
      AND total_value > po_approval_threshold
   THEN return TRUE

2. Regula 2 — Cantitate maximă per produs:
   productIds = items.pluck('woo_product_id').filter().unique()
   IF productIds.isEmpty() return FALSE

   // Un singur query pentru toate produsele (evită N+1):
   productSupplierMap = ProductSupplier
     .where('supplier_id', this.supplier_id)
     .whereIn('woo_product_id', productIds)
     .keyBy('woo_product_id')

   FOREACH item IN items:
     ps = productSupplierMap.get(item.woo_product_id)
     IF ps AND ps.po_max_qty AND item.quantity > ps.po_max_qty
     THEN return TRUE

3. return FALSE

Algoritmul FIFO de alocare cantități

Implementat în CreatePurchaseOrder::afterCreate(). Rulează în DB::transaction() cu lockForUpdate() pentru a preveni race conditions la creare simultană de PO-uri.

Pregătire date (buildItemsForSupplier)

// Adunarea necesarelor pentru un furnizor:
requestItems = PurchaseRequestItem
  .where('supplier_id', supplierId)
  .where('status', 'pending')
  .whereHas('purchaseRequest', status IN ['submitted', 'partially_ordered'])
  .whereRaw('quantity > COALESCE(ordered_quantity, 0)')  // exclude complet comandate
  .orderByDesc('is_urgent')
  .orderBy('needed_by')

// Grupare după produs (woo_product_id sau product_name):
groups = requestItems.groupBy(fn => woo_id ? 'woo:{id}' : 'name:{name}')

// Per grup → un PO item cu:
//   quantity = SUM(remaining) per item din grup
//   sources_json = array cu {request_item_id, quantity(remaining), is_urgent, needed_by, ...}

Execuție FIFO (afterCreate)

FOREACH orderItem IN purchaseOrder.items:
  sources = JSON.parse(orderItem.sources_json)

  // Sortare FIFO:
  // 1. Urgente (is_urgent=true) primele
  // 2. În caz de egalitate: needed_by ASC (cel mai vechi termen)
  sources.sortBy(b.is_urgent DESC, a.needed_by ASC)

  remaining = (float) orderItem.quantity
  updatedSources = []

  FOREACH source IN sources:
    requestItem = PurchaseRequestItem.lockForUpdate().find(source.request_item_id)

    alreadyOrdered  = requestItem.ordered_quantity
    originalQty     = requestItem.quantity
    availableInItem = MAX(0, originalQty - alreadyOrdered)

    // Alocăm minimul dintre: ce mai are PO de dat, ce vrea sursa, ce mai poate primi sursa
    canAllocate = MIN(remaining, source.quantity, availableInItem)

    IF canAllocate <= 0:
      updatedSources.push({...source, allocated_qty: 0})
      CONTINUE

    remaining -= canAllocate

    // Incrementare atomică ordered_quantity:
    PurchaseRequestItem.where('id', requestItem.id).increment('ordered_quantity', canAllocate)
    requestItem.refresh()

    // Dacă item complet acoperit → STATUS_ORDERED:
    IF requestItem.isFullyOrdered():  // ordered_quantity >= quantity
      requestItem.update({status: 'ordered', purchase_order_item_id: orderItem.id})
    ELSE:
      requestItem.update({purchase_order_item_id: orderItem.id})  // rămâne pending

    updatedSources.push({...source, allocated_qty: canAllocate})

  // Salvare sources_json cu allocated_qty completate:
  orderItem.updateQuietly({sources_json: JSON.stringify(updatedSources), ...})

// Post-transaction: recalculateStatus() pe fiecare PurchaseRequest afectat
FOREACH requestId IN UNIQUE(affectedRequestIds):
  PurchaseRequest.find(requestId).recalculateStatus()

Reversionarea itemelor la REJECTED/CANCELLED

La tranziția PO → rejected sau PO → cancelled, sistemul apelează revertRequestItemsToPending():

// Pentru fiecare item din PO:
FOREACH orderItem IN purchaseOrder.items:
  sources = JSON.parse(orderItem.sources_json ?? '[]')

  FOREACH source IN sources:
    allocatedQty = source.allocated_qty ?? 0
    IF allocatedQty <= 0: CONTINUE

    requestItem = PurchaseRequestItem.find(source.request_item_id)
    IF NOT requestItem: CONTINUE

    // Decrementare ordered_quantity:
    newOrdered = MAX(0, requestItem.ordered_quantity - allocatedQty)
    requestItem.update({
      ordered_quantity: newOrdered,
      status: 'pending',                // revine la pending
      purchase_order_item_id: null      // deconectare de la PO item
    })

    affectedRequestIds.add(requestItem.purchase_request_id)

// Recalculare status pentru fiecare PNR afectat:
FOREACH requestId IN affectedRequestIds:
  PurchaseRequest.find(requestId).recalculateStatus()

Auto-generare PNR din comenzi WooCommerce

// PurchaseRequest::createFromWooOrder(WooOrder $order): ?self

1. Verifică dacă există deja un PNR pentru această comandă (idempotent)
2. Filtrează itemele comenzii cu procurement_type = 'on_demand'
3. Dacă nu există → return null
4. Creează PNR cu status = submitted (nu draft) și source_type = 'woo_order'
5. Creează câte un PurchaseRequestItem per produs on_demand:
   - is_urgent = true
   - client_reference = "Comandă WooCommerce #{order.number}"
