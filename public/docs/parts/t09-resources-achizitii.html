2.2 Pre-populare din BuyerDashboard (logică specială)

Când CreatePurchaseOrder primește query param ?supplier_id=X (link din BuyerDashboardPage sau UnassignedItemsPage), metoda fillForm() apelează buildItemsForSupplier($supplierId) care:

  1. Preia toate PurchaseRequestItem cu supplier_id = X, status pending, din necesare cu status submitted sau partially_ordered, unde quantity > ordered_quantity.
  2. Sortează: urgent întâi, apoi după needed_by crescător (FIFO).
  3. Grupează pe produs (woo_product_id sau product_name).
  4. Calculează cantitatea totală rămasă (= quantity - ordered_quantity sumată pe grup).
  5. Auto-completează supplier_sku și unit_price din ProductSupplier.
  6. Construiește sources_json cu metadatele fiecărei surse (consultant, locație, dată, client, etc.).

2.3 Alocare FIFO după creare (afterCreate)

Executat în DB::transaction() cu lockForUpdate() (previne race condition la creare simultană):

  1. Pentru fiecare PurchaseOrderItem cu sources_json nevid, sortează sursele FIFO (urgent first → needed_by asc).
  2. Iterează sursele și alocă din cantitatea disponibilă a PO (remaining): canAllocate = min(remaining, sourceQty, availableInItem).
  3. Incrementează ordered_quantity pe PurchaseRequestItem.
  4. Dacă item-ul de necesar este complet acoperit (isFullyOrdered()) → status ordered + setează purchase_order_item_id.
  5. Dacă parțial → rămâne pending cu referința la PO setată.
  6. Salvează allocated_qty per sursă în sources_json.
  7. La final: recalculateStatus() pe toate necesarele afectate.

2.9 Logica markRequestItemsAsOrdered

Apelată când PO trece la status sent. Parcurge toate PurchaseOrderItem-urile, extrage toți request_item_id cu allocated_qty > 0 din sources_json (fallback pe FK primar). Marchează ca ordered doar itemele de necesar care sunt complet acoperite (isFullyOrdered()). Apelează recalculateStatus() pe necesarele afectate.

2.10 Logica revertShortfallToRequestItems (recepție cu lipsuri)

La recepție parțială: pentru fiecare linie cu lipsuri, parcurge sources_json în ordine inversă FIFO (ultimii alocați — primii returnați). Scade din ordered_quantity al itemului de necesar și resetează statusul la pending. Dacă ordered_quantity ajunge la 0 → șterge și referința la PO.

5. Rezumat flux Achiziții

Consultant → Creează PurchaseRequest (draft)
           → Trimite (submitted) → notificare buyers
Buyer      → Creează PurchaseOrder (?supplier_id=X din Dashboard)
             ↑ pre-populate FIFO din necesare pending
           → Plasează comanda
             ↑ dacă depășește plafon/max-qty → pending_approval
             ↑ altfel → approved automat
Manager    → Aprobă / Respinge
             ↑ Respingere → revertRequestItemsToPending, notifică buyer
Buyer      → Trimite email furnizor / Marchează trimis
             ↑ → sent + markRequestItemsAsOrdered()
Buyer      → Recepție marfă (cu cantități per linie)
             ↑ lipsuri → revertShortfallToRequestItems (inv. FIFO)
             ↑ → received + notifică consultanții cu lipsuri
