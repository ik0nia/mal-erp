2. Documentație detaliată per job

BaseAiJob

Fișier: app/Jobs/BaseAiJob.php

Clasă abstractă. Nu se dispatchează direct. Toate job-urile AI din Social Media și unele din email extind această clasă.

Metode utilitare oferite subclaselor:

  • getApiKey(): string — citește cheia Anthropic din AppSetting (criptat), cu fallback pe env('ANTHROPIC_API_KEY'); aruncă RuntimeException dacă lipsește.
  • recordUsage(type, model, inputTokens, outputTokens, context) — înregistrează consumul AI în tabelul ai_usage_logs prin AiUsageLog::record(); erorile la scriere sunt doar logate (nu opresc job-ul).
  • failed(Throwable) — handler implicit: loghează eroarea în Log::error cu clasa concretă și stack trace.

Defaults moștenite:

  • $tries = 2, $timeout = 120, $backoff = 30 (suprascribile în subclase)

Condiții de eșec:

Prin sine nu eșuează — erorile reale sunt în subclase.

Efecte secundare:

Scriere în ai_usage_logs (opțional, prin subclase).

FetchEmailsJob

Fișier: app/Jobs/FetchEmailsJob.php

Constructor (parametri):

Fără parametri. Configurația IMAP se citește din AppSetting la runtime.

Ce face handle():

  1. Citește credențialele IMAP (KEY_IMAP_HOST, KEY_IMAP_PORT, KEY_IMAP_ENCRYPTION, KEY_IMAP_USERNAME, KEY_IMAP_PASSWORD); dacă lipsesc, iese silențios.
  2. Construiește două index-uri pentru auto-asociere: suppliers.email → supplier_id și supplier_contacts.email → supplier_id; plus email → contact_id.
  3. Calculează data de căutare: SINCE N-3 zile (toleranță la scheduler oprit max. 3 zile).
  4. Obține lista tuturor folder-elor IMAP (recursiv), excluzând trash, spam, junk, drafts.
  5. Pentru fiecare folder: reconectare IMAP proaspătă (prevenție scurgeri de memorie), apoi:
    • PASS 1: UID SEARCH SINCE <data> — returnează lista de UID-uri (nu face FETCH ALL).
    • Filtrare UID-uri deja existente în email_messages (după imap_folder + imap_uid).
    • PASS 2: Fetch individual al fiecărui UID nou; extrage: from, subject (decode MIME), to, cc, atașamente (doar metadata), body_html, body_text, sent_at.
    • Rezolvare furnizor: mai întâi după suppliers.email, apoi după supplier_contacts.email.
    • Creare EmailMessage în DB (cu leaveUnread() — nu marchează ca citit).
  6. Loghează totalul emailurilor noi importate.

Condiții de eșec:

  • Credențiale IMAP lipsă → exit silențios (nu fail).
  • Eroare SEARCH pe un folder → skip folder (warning în log), continuă cu următorul.
  • Eroare Duplicate entry la INSERT → ignorată silențios (race condition la re-run).
  • $tries = 1 — nu re-încearcă: dacă serverul IMAP e down, se re-rulează la cronul de 5 min.

Efecte secundare (DB writes):

  • INSERT în email_messages (cu supplier_id și supplier_contact_id dacă sunt rezolvați).
  • Nu dispatchează ProcessEmailAIJob (codul există, dar e comentat — dezactivat temporar).

ProcessEmailAIJob

Fișier: app/Jobs/ProcessEmailAIJob.php

Constructor (parametri):

  • int $emailId — ID-ul înregistrării din email_messages

Ce face handle():

  1. Încarcă emailul cu relațiile supplier și supplierContact; dacă nu există sau e deja procesat (agent_processed_at not null), iese.
  2. Verifică prezența KEY_ANTHROPIC_API_KEY; dacă lipsește, skip cu warning.
  3. Construiește textul emailului: subiect + expeditor + corp text (preferat) sau HTML stripped, maxim 4000 caractere; adaugă lista de atașamente.
  4. Apelează Claude Haiku (max_tokens: 1024) cu un prompt care cere JSON structurat: type, summary, urgency, needs_reply, sentiment, products_mentioned, prices_mentioned, invoice_number, delivery_date, payment_terms, discount_mentioned, action_items, key_info.
  5. Parsează răspunsul JSON (curăță eventualul markdown ```json```).
  6. Salvează agent_actions (JSON complet) și agent_processed_at = now() pe email.
  7. Salvează entitățile în email_entities: prețuri (tip product_price), dată livrare, număr factură, termeni plată, reducere.
  8. Salvează cotațiile de preț în supplier_price_quotes: doar dacă emailul are supplier_id și tipul nu e automated/general; încearcă să potrivească produsul cu catalogul WooCommerce prin matchProduct().
  9. La excepție: marchează emailul cu agent_processed_at = now() și agent_actions = {error: ...} (previne retry-uri infinite), then rethrows.

Condiții de eșec:

  • API Anthropic down sau status non-2xx → returnează null (nu aruncă), emailul rămâne neprocesat.
  • JSON invalid în răspuns Claude → warning în log, returnează null.
  • Excepție generală → marchează emailul procesat cu eroare și rethrows pentru retry (max 2).

Efecte secundare (DB writes):

  • UPDATE email_messages (agent_actions, agent_processed_at).
  • INSERT în email_entities (prețuri, date livrare, facturi, termeni plată, reduceri).
  • INSERT în supplier_price_quotes.
  • INSERT în ai_usage_logs via AiUsageLog::record().

Metodă auxiliară publică: matchProduct(string $rawName, ?string $sku)

Algoritm de potrivire produs din email cu catalogul WooCommerce. Strategii în ordine:

  1. Potrivire exactă după SKU (dacă AI a extras codul explicit, ≥4 caractere).
  2. Potrivire exactă după LOWER(name).
  3. Scoring pe cuvinte distinctive (≥4 litere, fără stopwords, non-numerice): calculează câte cuvinte din rawName apar în titlul candidaților; prag minim 60% sau bonus +10 pentru potrivire completă a frazei.

GenerateBiAnalysisJob

Fișier: app/Jobs/GenerateBiAnalysisJob.php

Constructor (parametri):

  • int $analysisId — ID înregistrare bi_analyses
  • string $sections — 'both' | 'stock' | 'online' (default: 'both')
  • Carbon $dateFrom — start perioadă (default: 30 zile în urmă)
  • Carbon $dateTo — end perioadă (default: acum)

Ce face handle():

  1. Încarcă BiAnalysis sau aruncă 404.
  2. Verifică ANTHROPIC_API_KEY din config (nu AppSetting — diferit față de celelalte job-uri AI).
  3. Adună metrici (gatherMetrics()):
    • Stoc: daily_stock_metrics groupat zilnic/săptămânal/lunar în funcție de interval (≤30 / ≤180 / >180 zile); top 15 produse cu scăderi stoc; număr produse în stoc/fără stoc; modificări prețuri cu top 10 creșteri și top 10 scăderi. Filtrare exclusivă pe product_type = 'shop'.
    • Online: woo_orders groupate la aceeași granularitate; breakdown pe status; top 15 produse vândute din woo_order_items.
    • Context precedent (singur zi): ultimele 30 de zile din daily_stock_metrics și comenzi.
  4. Adună context (fetchPastAnalyses()): rapoarte BI anterioare din perioada curentă (max 2 lunar, max 4 săptămânal, max 2 manual) — truncate la 3000/2500/1500 caractere.
  5. Construiește prompt (buildPrompt()): context business Malinco (canal principal fizic, problema produse voluminoase online), date stoc + prețuri + comenzi, instrucțiuni raport în română (secțiuni diferite per $sections), inclusiv recomandări aprovizionare și sezonalitate.
  6. Apelează Claude Sonnet (max_tokens: 12000).
  7. Calculează costul estimat USD (3$/M input + 15$/M output).
  8. Salvează content, metrics_snapshot (cu tokens și cost), status = 'done', generated_at.
  9. La excepție: status = 'failed' + error_message.

Condiții de eșec:

  • ANTHROPIC_API_KEY lipsă → RuntimeException → status failed.
  • Claude returnează conținut gol → RuntimeException → status failed.
  • $tries = 1 — fără retry.

Efecte secundare (DB writes):

  • UPDATE bi_analyses (content, metrics_snapshot, status, generated_at).
  • INSERT în ai_usage_logs.

GenerateChatSummaryJob

Fișier: app/Jobs/GenerateChatSummaryJob.php

Constructor (parametri):

  • string $sessionId — ID-ul sesiunii de chat

Ce face handle():

  1. Caută ChatContact după session_id; dacă nu există, iese.
  2. Încarcă ultimele 30 de mesaje din chat_logs (ordonat cronologic).
  3. Dacă sunt mai puțin de 2 mesaje, iese.
  4. Verifică KEY_ANTHROPIC_API_KEY.
  5. Formatează conversația: fiecare mesaj truncat la 250 caractere, prefix Client/Bot.
  6. Apelează Claude Haiku (max_tokens: 200) cu prompt care cere JSON: {"interested_in": "...", "summary": "..."}.
  7. Actualizează ChatContact cu summary și interested_in.
  8. Înregistrează consumul în ai_usage_logs.

Condiții de eșec:

  • API error → warning în log, iese silențios (nu rethrows).
  • JSON invalid → iese silențios.

Efecte secundare (DB writes):

  • UPDATE chat_contacts (summary, interested_in).
  • INSERT în ai_usage_logs.

ImportWooCategoriesJob

Fișier: app/Jobs/ImportWooCategoriesJob.php

  1. Încarcă conexiunea (findOrFail).

PushWinmentorPricesToWooJob

Fișier: app/Jobs/PushWinmentorPricesToWooJob.php

Constructor (parametri):

  • int $syncRunId
  • int $wooConnectionId
  • array $updates — lista de {row, sku, woo_id, regular_price}

Ce face handle():

  1. Verifică că SyncRun există și nu e cancelat; altfel iese.
  2. Verifică că IntegrationConnection este WooCommerce activă.
  3. Construiește payload batch și apelează WooClient::updateProductPricesBatch().
  4. La eșec batch: fallback la actualizare individuală produs cu produs (updateProductPrice()).
  5. Înregistrează progresul în SyncRun (cu lockForUpdate în tranzacție): contoare site_price_updates, site_price_update_failures, site_price_push_queued.
  6. Dacă site_price_push_queued ajunge la 0: finalizează run-ul (success sau failed dacă au existat erori).

Condiții de eșec:

  • Conexiune WooCommerce invalidă → înregistrează eșecul și iese (fără retry).
  • Erori API pe produs individual → acumulate în newErrors, continuat cu restul.
  • La eșec definitiv job (failed()): înregistrează toate update-urile ca eșuate în SyncRun.

Efecte secundare (DB writes):

  • UPDATE prețuri pe WooCommerce via REST API.
  • UPDATE sync_runs.stats și sync_runs.errors (atomic cu lockForUpdate).
  • UPDATE sync_runs.status și finished_at la finalizare.

GenerateSocialPostJob

Fișier: app/Jobs/GenerateSocialPostJob.php

Constructor (parametri):

  • int $postId — ID din social_posts

Ce face handle():

  1. Încarcă postarea cu relațiile product, account, brand.
  2. Setează status generating pe postare.
  3. Generare caption: SocialCaptionService::generate($post, $styleProfile) cu Claude AI — returnează caption, hashtags, image_prompt, graphic_title, graphic_subtitle.
  4. Rezolvă titlul graficii: din răspunsul AI sau fallback la numele produsului (fără codul de catalog) sau brand.
  5. Rezolvă subtitle: prima linie din caption (fără emoji leading), trunchiat la 70 de caractere.
  6. Rezolvă label-ul: bazat pe brief_type (brand, promo, product).
  7. Salvează câmpurile generate pe postare.
  8. Randare imagine: NodeImageRenderer::render() — apelează serviciul Node.js canvas local cu imaginea produsului (URL), logo brand (local dacă există), titlu, subtitle, label și config template.
  9. Salvează image_path și setează status ready.
  10. La eșec: status failed + error_message.

Rezolvare template:

Caută template în ordine: slug explicit → template activ cu același layout → template default (GraphicTemplate::default()).

Condiții de eșec:

  • Claude API down → status failed.
  • Node.js renderer down sau timeout → status failed.
  • $tries = 2, $backoff = 15 s.

Efecte secundare (DB writes):

  • UPDATE social_posts (status, caption, hashtags, image_prompt, graphic_title, graphic_subtitle, graphic_label, image_path).

  2. Setează status generating.
  3. Dacă graphic_title e gol: apelează SocialCaptionService::generateGraphicTextsOnly() pentru a obține titlu+subtitle; la eșec, fallback la brand/produs.
  4. Dacă graphic_title există deja: îl folosește direct.
  5. Apelează NodeImageRenderer::render() cu aceiași parametri ca în GenerateSocialPostJob.
  6. Salvează image_path și status ready.
  7. La eșec: status failed + error_message.

Diferență față de GenerateSocialPostJob: Nu regenerează caption/hashtag-uri — doar imaginea grafică.

Condiții de eșec:

  • Node.js renderer → status failed; $tries = 2.

Efecte secundare (DB writes):

  • UPDATE social_posts (status, image_path, opțional graphic_title, graphic_subtitle, graphic_label).

PublishSocialPostJob

Fișier: app/Jobs/PublishSocialPostJob.php

Constructor (parametri):

  • int $postId — ID din social_posts

Ce face handle():

  1. Deschide tranzacție DB cu lockForUpdate pe postare — protecție anti-dublă-publicare.
  2. Dacă postarea e deja published → iese din tranzacție.
  3. Dacă postarea nu e scheduled → warning în log, iese.
  4. Setează status publishing.
  5. Obține token Meta pentru contul asociat.
  6. Apelează MetaGraphClient::publishPost($account_id, $message, $imageUrl) unde $message = $post->getFullCaption() (caption + hashtag-uri) și $imageUrl = $post->getImageUrl().
  7. Salvează platform_post_id, platform_url (URL Facebook), published_at, status published.

Condiții de eșec:

  • Token lipsă → RuntimeException → retry; $tries = 3, $backoff = 60 s.
  • Meta Graph API error → rethrown → retry.
  • La eșec definitiv (failed()): status failed + error_message pe postare.

Efecte secundare:

  • UPDATE social_posts (status, platform_post_id, platform_url, published_at).
  • Publicare pe Facebook Page via Meta Graph API (efect extern permanent).

Note despre schedule

  • Comenzile BI pentru rapoarte cu AI (bi:generate-*) folosesc Claude Sonnet și pot dura câteva minute — de aceea sunt marcate cu runInBackground.
  • bi:compute-daily este idempotent (poate fi rulat de mai multe ori fără efecte negative), de aceea nu are withoutOverlapping.
  • Procesarea AI a emailurilor și redescoperirea contactelor sunt dezactivate temporar din motive de consum API; pot fi reactivate prin decomentarea liniilor din schedule.
  • Rapoartele trimestriale și semestriale sunt definite ca monthly în Laravel Scheduler cu o condiție when() suplimentară pentru a rula doar în lunile corecte.
