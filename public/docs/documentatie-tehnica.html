4. Algoritmul FIFO de alocare cantități (afterCreate)

Când un PurchaseOrder este creat, metoda afterCreate() din CreatePurchaseOrder leagă fiecare PurchaseOrderItem de sursele sale (PurchaseRequestItem-uri) din sources_json. Algoritmul alocă cantitățile parțial sau complet, respectând ordinea FIFO (urgente întâi, apoi data limită crescătoare).

4.1 Preprocesare — buildItemsForSupplier()

Înainte de creare, dacă PO-ul este inițiat din BuyerDashboard cu parametrul ?supplier_id=X, formularul este pre-populat de buildItemsForSupplier():

  1. Selectează toate PurchaseRequestItem-urile cu:
    • supplier_id = $supplierId
    • status = pending
    • PNR-ul părinte în status submitted sau partially_ordered
    • quantity > COALESCE(ordered_quantity, 0) (exclude complet comandate)
  2. Sortează: is_urgent DESC, needed_by ASC (FIFO).
  3. Grupează după woo_product_id (sau product_name pentru produse fără WooID).
  4. Pentru fiecare grup: sumează cantitățile rămase (quantity - ordered_quantity), construiește un array sources cu câmpurile: request_item_id, request_number, consultant, location, quantity (cantitate rămasă), is_urgent, needed_by, client_reference, allocated_qty=0.0.
  5. Serialzează sources ca JSON în câmpul sources_json al formularului.

4.2 Execuția afterCreate() — pas cu pas

Întreaga logică este învelită în DB::transaction() cu lockForUpdate() pe fiecare PurchaseRequestItem (previne race condition la creare simultană de PO-uri). 

// Pseudocod simplificat al afterCreate()
foreach ($this->record->items as $orderItem) {
    $sources = json_decode($orderItem->sources_json);  // array de surse

    // 1. FIFO sort: urgent first, then earliest needed_by
    usort($sources, fn($a, $b) =>
        ($b['is_urgent'] - $a['is_urgent'])
        ?: strcmp($a['needed_by'], $b['needed_by'])
    );

    $remaining = (float) $orderItem->quantity;  // cantitatea totală a liniei PO

    foreach ($sources as $source) {
        $requestItem = PurchaseRequestItem::lockForUpdate()->find($source['request_item_id']);

        // Calculăm cât putem aloca din această sursă
        $alreadyOrdered  = $requestItem->ordered_quantity;
        $availableInItem = $requestItem->quantity - $alreadyOrdered;  // max disponibil în sursă
        $canAllocate     = min($remaining, $source['quantity'], $availableInItem);

        if ($canAllocate <= 0) { ... skip ... }

        $remaining -= $canAllocate;

        // Incrementare atomică ordered_quantity
        PurchaseRequestItem::where('id', $requestItem->id)
            ->increment('ordered_quantity', $canAllocate);

        $requestItem->refresh();  // actualizăm valorile post-increment

        if ($requestItem->isFullyOrdered()) {
            // Item complet acoperit → STATUS_ORDERED
            $requestItem->update([
                'status'                 => PurchaseRequestItem::STATUS_ORDERED,
                'purchase_order_item_id' => $orderItem->id,
            ]);
        } else {
            // Item parțial comandat → rămâne PENDING, dar cu referința PO
            $requestItem->update([
                'purchase_order_item_id' => $orderItem->id,
            ]);
        }

        $source['allocated_qty'] = $canAllocate;  // înregistrăm cât s-a alocat
    }

    // Salvăm sources_json cu allocated_qty completat
    $orderItem->updateQuietly([
        'sources_json'             => json_encode($updatedSources),

4.5 Protecție race condition

Crearea simultană a două PO-uri pentru același furnizor ar putea supra-aloca cantitățile. Protecția se face pe trei niveluri:

  1. DB::transaction() — toate incrementările și actualizările sunt atomice.
  2. lockForUpdate() pe PurchaseRequestItem — blochează rândul DB pentru citire și scriere până la finalul tranzacției.
  3. Recalcul post-lock — alreadyOrdered se citește după obținerea lock-ului ($requestItem->ordered_quantity la momentul lock-ului), nu din datele formularului.
  4. firstOrCreate() în getOrCreateDraft() — previne duplicate de drafturi per user.

5. Logica needsApproval()

Metoda PurchaseOrder::needsApproval() este apelată în acțiunea "Plasează comanda" pentru a decide dacă PO-ul se aprobă automat (→ approved) sau necesită aprobare manuală (→ pending_approval). 

public function needsApproval(): bool
{
    $this->loadMissing(['supplier', 'items']);

    // Regula 1: valoare totală PO > plafon furnizor
    if ($this->supplier->po_approval_threshold &&
        (float) $this->total_value > (float) $this->supplier->po_approval_threshold) {
        return true;
    }

    // Regula 2: oricare item depășește cantitatea maximă per produs
    $productIds = $this->items->pluck('woo_product_id')->filter()->unique()->values();

    if ($productIds->isEmpty()) {
        return false;
    }

    // Query unic pentru toate ProductSupplier-urile relevante (evitare N+1)
    $productSupplierMap = ProductSupplier::query()
        ->where('supplier_id', $this->supplier_id)
        ->whereIn('woo_product_id', $productIds)
        ->get(['woo_product_id', 'po_max_qty'])
        ->keyBy('woo_product_id');

    foreach ($this->items as $item) {
        if (! $item->woo_product_id) continue;

        $ps = $productSupplierMap->get($item->woo_product_id);

        if ($ps && $ps->po_max_qty && (float) $item->quantity > (float) $ps->po_max_qty) {
            return true;
        }
    }

    return false;
}

6. Algoritmul recalculateStatus() pe PurchaseRequest

Metoda PurchaseRequest::recalculateStatus() este apelată de fiecare dată când starea unui PurchaseRequestItem sau ordered_quantity se modifică. Folosește saveQuietly() pentru a evita declanșarea evenimentelor Eloquent recurente. 

public function recalculateStatus(): void
{
    $items = $this->items()->get(['status', 'quantity', 'ordered_quantity']);

    // Guard: fără items → nu face nimic
    if ($items->isEmpty()) {
        return;
    }

    // Excludem itemii anulați din calcul
    $active = $items->where('status', '!=', PurchaseRequestItem::STATUS_CANCELLED);

    // Branch 1: toți itemii sunt anulați
    if ($active->isEmpty()) {
        $this->forceFill(['status' => self::STATUS_CANCELLED])->saveQuietly();
        return;
    }

    $total = $active->count();

    // Un item "complet comandat" = ordered_quantity >= quantity
    $fullyOrdered = $active->filter(
        fn ($i) => (float) $i->ordered_quantity >= (float) $i->quantity
    )->count();

    // "Are cel puțin un item parțial comandat" = ordered_quantity > 0
    $hasAnyOrdered = $active->filter(
        fn ($i) => (float) $i->ordered_quantity > 0
    )->count() > 0;

    // Branch 2: toți itemii activi sunt complet acoperiți
    if ($fullyOrdered >= $total) {
        $this->forceFill(['status' => self::STATUS_FULLY_ORDERED])->saveQuietly();

    // Branch 3: cel puțin un item are ordered_quantity > 0 (dar nu toți sunt complet)
    } elseif ($hasAnyOrdered) {
        $this->forceFill(['status' => self::STATUS_PARTIALLY_ORDERED])->saveQuietly();

    // Branch 4: niciun item nu are cantitate comandată → revenire la submitted
    } else {
        $this->forceFill(['status' => self::STATUS_SUBMITTED])->saveQuietly();
    }
}

6.1 Arbore de decizie

Condiție evaluată Status rezultat Cazuri tipice
$items.isEmpty() Fără modificare (return timpuriu) PNR fără niciun item (caz degenerat)
$active.isEmpty() (toți itemii = cancelled) cancelled Consultantul a anulat manual toți itemii
$fullyOrdered >= $total (toți activi: ordered_qty ≥ qty) fully_ordered PO creat și trimis acoperă tot necesarul
$hasAnyOrdered = true (cel puțin un item parțial sau complet acoperit) partially_ordered PO acoperă o parte din itemi/cantități; restul rămâne în așteptare
Niciuna din cele de mai sus (toți activi cu ordered_qty = 0) submitted PO anulat/respins → toate alocările revenite; PNR reintră în coadă

6.2 Contexte de apel

  • afterCreate() din CreatePurchaseOrder — după alocarea FIFO, pentru fiecare PNR afectat
  • markRequestItemsAsOrdered() — la tranzitia PO la sent
  • revertRequestItemsToPending() — la CANCELLED sau REJECTED al PO-ului
  • revertShortfallToRequestItems() — la recepție parțială cu lipsuri
  • cancel_item action din infolist PurchaseRequest — la anularea manuală a unui item

FetchEmailsJob

Fișier: app/Jobs/FetchEmailsJob.php

Constructor (parametri):

Fără parametri. Configurația IMAP se citește din AppSetting la runtime.

Ce face handle():

  1. Citește credențialele IMAP (KEY_IMAP_HOST, KEY_IMAP_PORT, KEY_IMAP_ENCRYPTION, KEY_IMAP_USERNAME, KEY_IMAP_PASSWORD); dacă lipsesc, iese silențios.
  2. Construiește două index-uri pentru auto-asociere: suppliers.email → supplier_id și supplier_contacts.email → supplier_id; plus email → contact_id.
  3. Calculează data de căutare: SINCE N-3 zile (toleranță la scheduler oprit max. 3 zile).
  4. Obține lista tuturor folder-elor IMAP (recursiv), excluzând trash, spam, junk, drafts.
  5. Pentru fiecare folder: reconectare IMAP proaspătă (prevenție scurgeri de memorie), apoi:
    • PASS 1: UID SEARCH SINCE <data> — returnează lista de UID-uri (nu face FETCH ALL).
    • Filtrare UID-uri deja existente în email_messages (după imap_folder + imap_uid).
    • PASS 2: Fetch individual al fiecărui UID nou; extrage: from, subject (decode MIME), to, cc, atașamente (doar metadata), body_html, body_text, sent_at.
    • Rezolvare furnizor: mai întâi după suppliers.email, apoi după supplier_contacts.email.
    • Creare EmailMessage în DB (cu leaveUnread() — nu marchează ca citit).
  6. Loghează totalul emailurilor noi importate.

Condiții de eșec:

  • Credențiale IMAP lipsă → exit silențios (nu fail).
  • Eroare SEARCH pe un folder → skip folder (warning în log), continuă cu următorul.
  • Eroare Duplicate entry la INSERT → ignorată silențios (race condition la re-run).
  • $tries = 1 — nu re-încearcă: dacă serverul IMAP e down, se re-rulează la cronul de 5 min.

Efecte secundare (DB writes):

  • INSERT în email_messages (cu supplier_id și supplier_contact_id dacă sunt rezolvați).
  • Nu dispatchează ProcessEmailAIJob (codul există, dar e comentat — dezactivat temporar).

ProcessEmailAIJob

Fișier: app/Jobs/ProcessEmailAIJob.php

Constructor (parametri):

  • int $emailId — ID-ul înregistrării din email_messages

Ce face handle():

  1. Încarcă emailul cu relațiile supplier și supplierContact; dacă nu există sau e deja procesat (agent_processed_at not null), iese.
  2. Verifică prezența KEY_ANTHROPIC_API_KEY; dacă lipsește, skip cu warning.
  3. Construiește textul emailului: subiect + expeditor + corp text (preferat) sau HTML stripped, maxim 4000 caractere; adaugă lista de atașamente.
  4. Apelează Claude Haiku (max_tokens: 1024) cu un prompt care cere JSON structurat: type, summary, urgency, needs_reply, sentiment, products_mentioned, prices_mentioned, invoice_number, delivery_date, payment_terms, discount_mentioned, action_items, key_info.
  5. Parsează răspunsul JSON (curăță eventualul markdown ```json```).
  6. Salvează agent_actions (JSON complet) și agent_processed_at = now() pe email.
  7. Salvează entitățile în email_entities: prețuri (tip product_price), dată livrare, număr factură, termeni plată, reducere.
  8. Salvează cotațiile de preț în supplier_price_quotes: doar dacă emailul are supplier_id și tipul nu e automated/general; încearcă să potrivească produsul cu catalogul WooCommerce prin matchProduct().
  9. La excepție: marchează emailul cu agent_processed_at = now() și agent_actions = {error: ...} (previne retry-uri infinite), then rethrows.

Condiții de eșec:

  • API Anthropic down sau status non-2xx → returnează null (nu aruncă), emailul rămâne neprocesat.
  • JSON invalid în răspuns Claude → warning în log, returnează null.
  • Excepție generală → marchează emailul procesat cu eroare și rethrows pentru retry (max 2).

Efecte secundare (DB writes):

  • UPDATE email_messages (agent_actions, agent_processed_at).
  • INSERT în email_entities (prețuri, date livrare, facturi, termeni plată, reduceri).
  • INSERT în supplier_price_quotes.
  • INSERT în ai_usage_logs via AiUsageLog::record().

Metodă auxiliară publică: matchProduct(string $rawName, ?string $sku)

Algoritm de potrivire produs din email cu catalogul WooCommerce. Strategii în ordine:

  1. Potrivire exactă după SKU (dacă AI a extras codul explicit, ≥4 caractere).
  2. Potrivire exactă după LOWER(name).
  3. Scoring pe cuvinte distinctive (≥4 litere, fără stopwords, non-numerice): calculează câte cuvinte din rawName apar în titlul candidaților; prag minim 60% sau bonus +10 pentru potrivire completă a frazei.

2.2 Pre-populare din BuyerDashboard (logică specială)

Când CreatePurchaseOrder primește query param ?supplier_id=X (link din BuyerDashboardPage sau UnassignedItemsPage), metoda fillForm() apelează buildItemsForSupplier($supplierId) care:

  1. Preia toate PurchaseRequestItem cu supplier_id = X, status pending, din necesare cu status submitted sau partially_ordered, unde quantity > ordered_quantity.
  2. Sortează: urgent întâi, apoi după needed_by crescător (FIFO).
  3. Grupează pe produs (woo_product_id sau product_name).
  4. Calculează cantitatea totală rămasă (= quantity - ordered_quantity sumată pe grup).
  5. Auto-completează supplier_sku și unit_price din ProductSupplier.
  6. Construiește sources_json cu metadatele fiecărei surse (consultant, locație, dată, client, etc.).

2.3 Alocare FIFO după creare (afterCreate)

Executat în DB::transaction() cu lockForUpdate() (previne race condition la creare simultană):

  1. Pentru fiecare PurchaseOrderItem cu sources_json nevid, sortează sursele FIFO (urgent first → needed_by asc).
  2. Iterează sursele și alocă din cantitatea disponibilă a PO (remaining): canAllocate = min(remaining, sourceQty, availableInItem).
  3. Incrementează ordered_quantity pe PurchaseRequestItem.
  4. Dacă item-ul de necesar este complet acoperit (isFullyOrdered()) → status ordered + setează purchase_order_item_id.
  5. Dacă parțial → rămâne pending cu referința la PO setată.
  6. Salvează allocated_qty per sursă în sources_json.
  7. La final: recalculateStatus() pe toate necesarele afectate.

2.9 Logica markRequestItemsAsOrdered

Apelată când PO trece la status sent. Parcurge toate PurchaseOrderItem-urile, extrage toți request_item_id cu allocated_qty > 0 din sources_json (fallback pe FK primar). Marchează ca ordered doar itemele de necesar care sunt complet acoperite (isFullyOrdered()). Apelează recalculateStatus() pe necesarele afectate.

2.10 Logica revertShortfallToRequestItems (recepție cu lipsuri)

La recepție parțială: pentru fiecare linie cu lipsuri, parcurge sources_json în ordine inversă FIFO (ultimii alocați — primii returnați). Scade din ordered_quantity al itemului de necesar și resetează statusul la pending. Dacă ordered_quantity ajunge la 0 → șterge și referința la PO.

           → Trimite (submitted) → notificare buyers

             ↑ pre-populate FIFO din necesare pending
           → Plasează comanda
             ↑ dacă depășește plafon/max-qty → pending_approval
             ↑ altfel → approved automat
Manager    → Aprobă / Respinge
             ↑ Respingere → revertRequestItemsToPending, notifică buyer
Buyer      → Trimite email furnizor / Marchează trimis
             ↑ → sent + markRequestItemsAsOrdered()
Buyer      → Recepție marfă (cu cantități per linie)
             ↑ lipsuri → revertShortfallToRequestItems (inv. FIFO)
             ↑ → received + notifică consultanții cu lipsuri
